Model 核心引擎

Model 是 Agent 的大脑。在 Agent Loop 中，每轮循环的核心动作就是调用 LLM API，传入当前 context（system prompt + 消息历史 + 工具定义），获取模型的下一步决策。

Agent Loop 中的位置

Agent Loop — Model 调用时机
1
context 管理（compaction、caching）
2
response = await callModel(messages, tools, options) ← Model 调用
3
解析 response：提取 text + tool_use blocks
4
如果有 tool_use → 执行工具 → push tool_result → continue loop
5
如果无 tool_use → 任务完成 → return result

流式调用机制

Agent 使用 SSE 流式接收模型输出，实现"边生成边处理"——工具调用可以在流式响应完成前就开始执行。

// Anthropic SDK 流式调用
const stream = client.messages.stream({
  model: "claude-sonnet-4-6-20250514",
  max_tokens: 16000,
  system: systemPrompt,
  messages: messages,
  tools: toolDefinitions,
});

for await (const event of stream) {
  if (event.type === 'content_block_delta') {
    // 文本增量 → 实时显示给用户
    emitToUI(event.delta.text);
  }
  if (event.type === 'content_block_start' && event.content_block.type === 'tool_use') {
    // 工具调用开始 → StreamingToolExecutor 提前启动
    startToolEarly(event.content_block);
  }
}

多模型路由

Gateway 模式根据模型名称前缀路由到不同 API 后端：

// gateway.ts — 模型路由
function routeModel(model) {
  if (model.startsWith('claude-'))
    return anthropicClient;  // Anthropic SDK
  if (model.startsWith('deepseek-'))
    return openaiClient({ baseURL: 'https://api.deepseek.com' });
  if (model.startsWith('gpt-'))
    return openaiClient;  // OpenAI SDK
}

System Prompt 指令装配

System Prompt 是 Agent 的"宪法"——定义身份、能力、约束和行为规范。它不是一段静态文本，而是在每次 Agent Loop 启动时动态组装的。

Agent Loop 中的位置

System Prompt 组装时机
0
Agent Loop 启动前 → buildSystemPrompt(context)
1
加载基础指令 + 环境信息 + git 状态
2
加载 WZXCLAW.md / CLAUDE.md（项目级 + 用户级）
3
注入 MCP 工具描述 + Skills 描述
4
插入 CACHE_BOUNDARY 标记（静态/动态分割）

组装流程

// system-prompt-builder.ts
function buildSystemPrompt(ctx) {
  let prompt = '';

  // [1] 基础指令：身份 + 能力 + 约束
  prompt += BASE_INSTRUCTIONS;     // ~8K tokens, 可缓存

  // [2] 环境信息：OS、shell、CWD、git branch
  prompt += formatEnvironment(ctx);

  // [3] WZXCLAW.md 记忆文件（8 级加载链）
  prompt += loadMemoryFiles(ctx.projectRoot);
  // → ~/.wzxclaw/WZXCLAW.md (global)
  // → project/WZXCLAW.md (project)
  // → project/.wzxclaw/WZXCLAW.md (local)

  // [4] Cache 边界 — 此前内容可被缓存
  prompt += '[CACHE_BOUNDARY]';

  // [5] 动态内容（每轮可能变化）
  prompt += formatMCPTools(ctx.mcpTools);
  prompt += formatSkills(ctx.skills);
  prompt += formatTaskContext(ctx.currentTask);

  return prompt;
}

Tools 行动能力

Tools 是 Agent 的"手"——让模型能读写文件、执行命令、搜索代码。没有 Tools，模型只能说话不能行动。

Agent Loop 中的位置

Tool 调度全流程
1
Model 返回 response → 解析出 tool_use blocks
2
StreamingToolExecutor：流式接收时就开始执行并发安全工具
3
partition(tools) → 并发安全组（并行） + 非安全组（串行）
4
PreToolUse hook → 权限检查 → tool.call(args) → PostToolUse hook
5
结果截断（maxResultSizeChars）→ 包装为 tool_result → push 到 messages

Tool 接口定义

// 每个工具必须实现的标准接口
interface Tool<Input, Output> {
  name: string;           // "Bash", "FileRead", "Grep"...
  description: string;    // 给模型看的说明
  inputSchema: JSONSchema; // Zod schema → JSON Schema

  call(args, ctx): Promise<ToolResult>;
  isConcurrencySafe(args): boolean;  // 能否并行
  isReadOnly(args): boolean;        // 只读？
  maxResultSizeChars: number;     // 结果截断上限
}

调度策略：并行 vs 串行

// streaming-tool-executor.ts
function partitionAndExecute(toolCalls) {
  const safe = toolCalls.filter(t => t.isConcurrencySafe);   // Read, Grep, Glob
  const unsafe = toolCalls.filter(t => !t.isConcurrencySafe); // Write, Edit, Bash

  // 并发安全工具 → Promise.allSettled 并行
  const safeResults = await Promise.allSettled(
    safe.map(t => executeWithHooks(t))
  );

  // 非并发安全工具 → for...of 串行
  const unsafeResults = [];
  for (const t of unsafe) {
    unsafeResults.push(await executeWithHooks(t));
  }

  return [...safeResults, ...unsafeResults];
}

MCP Servers 外部连接

MCP (Model Context Protocol) 是 AI 应用连接外部工具的标准化协议。它让工具集成从"每个自定义"变为"一次接入，处处可用"。

架构：Client-Server 模型

Agent Loop
tool_use → MCP 工具名匹配
→
MCP Client
callTool(name, args)
→
MCP Server
stdio / HTTP / SSE

连接生命周期

MCP 连接 → 发现 → 注入 → 调用
1
connect(serverConfig) → 启动子进程或 HTTP 连接
2
client.listTools() → 获取工具列表 (name + description + inputSchema)
3
包装为 Agent Tool 接口 → 注入到 system prompt 工具定义区
4
Agent Loop 中 → Model 返回 tool_use for MCP tool → client.callTool(name, args)
5
结果包装为 tool_result → 返回给 Model

代码层面：工具发现与注册

// 启动时连接所有 MCP Server
async function connectMCPServers(config) {
  for (const server of config.servers) {
    const client = await connect(server);
    const tools = await client.listTools();

    // 包装为标准 Tool 接口
    for (const tool of tools) {
      registry.register({
        name: `mcp_${server.name}_${tool.name}`,
        description: tool.description,
        inputSchema: tool.inputSchema,
        call: (args) => client.callTool(tool.name, args),
      });
    }
  }
}

Context 占用问题

每个 MCP Server 的工具 schema 全量注入 system prompt。连接越多 Server，context 越膨胀：

MCP Server 数工具定义占用占 200K 窗口
1~2K tokens1%
5~10K tokens5%
20~50K tokens25%

解决方案：Tool Search 动态加载 — 只在模型需要时才加载 MCP 工具定义，不预先全量注入。

Skills 渐进式上下文

Skills 是"按需展开的能力包"——注册时只占几十 token，触发时才注入完整指令。解决 MCP 工具定义膨胀 context 的核心问题。

注册 vs 展开：两个阶段

注册阶段（常驻 Context）

只有 name + description 注入到 Skill 工具定义中。
3 个 Skills 只占 ~65 tokens。

Context 隐形主线

Context 管理贯穿所有阶段。200K token 的窗口是硬约束——System Prompt、工具定义、消息历史、工具结果全部共享这一空间。Context 工程是 Agent 能否长时间运行的关键。

Agent Loop 中的位置

Context 管理发生在每次 Model 调用之前——确保输入不超过窗口限制：

// agent-loop.ts — 6 层 context 管理
while (!done) {
  messages = applyToolResultBudget(messages);    // [1] 截断大工具输出
  messages = applySnipCompact(messages);          // [2] 删除旧轮次
  messages = applyMicrocompact(messages);        // [3] 压缩工具密集区
  messages = applyContextCollapse(messages);      // [4] 细粒度折叠
  messages = autoCompactIfNeeded(messages);      // [5] 超阈值 → subagent 摘要

  response = await callModel(messages, tools);    // ← 管理后的 context

  if (response.error == 'prompt-too-long')
    messages = reactiveCompact(messages);          // [6] 紧急压缩
}

Auto-compact 详解（最重要的压缩层）

// autoCompact.ts — 超阈值时的核心压缩逻辑
async function autoCompactIfNeeded(messages) {
  const tokens = countTokens(messages);
  const threshold = MAX_CONTEXT - 13000; // 安全缓冲

  if (tokens > threshold) {
    // Fork 一个无工具的 subagent 来生成摘要
    const summary = await runForkedAgent({
      messages: messages,
      task: "Summarize this conversation preserving key facts",
      maxTurns: 1,
      tools: [],  // 不给工具，只做摘要
    });

    // 用摘要替换旧消息，保留最近几轮
    return [
      systemMsg(`[compacted] ${summary}`),
      ...recentMessages,
    ];
  }
  return messages;
}

Session vs Context Window（Managed Agents 架构）

Context Window（易失）

模型每轮"看到"的内容。有大小限制（200K）。Compaction 是不可逆的——压缩后原始信息从 context 永久消失（除非 Session 中有记录）。

Session Log（持久）

Append-only event log，记录所有交互（包括被 compaction 删除的）。通过 getEvents(pos) 可回溯任意时刻。Harness 可以 rewind 重读丢失的上下文。
