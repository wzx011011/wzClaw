Phase 12023
Model + System Prompt
Agent = Model + SystemPrompt

最原始的形态。往 Messages API 的 system 字段塞一段指令文本，模型按要求输出回答。模型能理解指令，但无法采取任何行动——纯文本输入，纯文本输出。

📋
Context 维度
Context = system_prompt + user_message。没有多轮，没有工具结果，context 即对话本身。模型看到的全部内容就是 prompt + 当前消息。无管理需求。
~2Ktokens/轮
解决的问题让模型按角色/风格/约束回答问题
核心局限"光说不做"——模型没有任何行动能力

Phase 22024.03 — 2024.12
+ Tool Use / Function Calling
Agent = Model + SystemPrompt + Tools

API 层面引入 tool_use 原语。模型不再只能说话，而是能发出结构化的工具调用请求。开发者负责执行工具并返回结果。从 Computer Use 到 Programmatic Tool Calling，Claude 逐步获得与数字世界交互的能力。

2024.03Claude 3 发布，Tool Use Beta 上线
2024.07扩展 tool use 文档和 structured output
2024.10Computer Use Beta — 首个控制桌面 GUI 的前沿模型
2024.12Programmatic Tool Calling — 并行执行、代码级编排
🔄
Context 维度
多轮对话出现，每轮追加 tool_use + tool_result。Context 开始线性膨胀——工具输出可能数千 token，很快逼近窗口上限。开发者被迫手动裁剪历史消息。200K 窗口成为天花板。
~20Ktokens/轮
解决的问题模型能行动了——调用函数、操作 GUI、执行代码
核心局限每个开发者重复造 agent loop、工具执行、重试逻辑

Phase 32024.11
+ MCP / Model Context Protocol
Agent = Model + Prompt + Tools + MCP

MCP 作为开源协议发布，定义了 AI 应用与外部工具之间的标准通信方式。从此不再需要为每个工具写自定义集成——只要实现 MCP Server，任何 AI 客户端都能发现和调用。这是从"硬编码集成"到"协议化连接"的范式转变。

2024.11.25MCP 开源协议发布（stdio / SSE 传输）
2025.Q1Streamable HTTP 传输协议，生产就绪
// MCP 统一了工具接入方式 AI App (MCP Client) ↔ MCP Protocol ↔ Tool Server // 支持动态发现工具，无需预加载 client.listTools() // 发现 client.callTool(name) // 调用
⚠️
Context 维度
Context 危机显现。每个 MCP Server 的工具定义（name + description + inputSchema）都要注入 system prompt。连接 20 个 MCP Server 可能消耗 30K–50K token，还没开始干活就占了窗口的 15–25%。
~50Ktokens 静态
解决的问题工具连接标准化，一次接入多处可用
核心局限加载 20 个 MCP Server 工具定义消耗数万 token，撑爆 context
3
Phase 42025.02 — 2025.10
Claude Code : Harness 产品化
Harness = Model + Prompt + Tools + MCP + Context Management

Claude Code 将散落的组件组装为完整的 agent runtime。它本身不是一个"功能"，而是一个精心调校的 harness——为 Claude 模型提供合适的工具、提示、上下文和权限控制。Context 工程在这个阶段正式成为核心学科。

2025.02.24Claude Code research preview 发布（随 Claude 3.7 Sonnet）
2025.05.25Claude 4 发布，Claude Code v1.0，Max 订阅计划
2025.06.12Plan Mode — 先规划再执行 (~v1.0.18)
2025.08.20/context 命令 — 可视化 context 占用 (v1.0.86)
Claude Code Harness = { model: "Claude 4 / Opus 4.5", system_prompt: "工具规范 + 代码风格 + git 工作流 + ...", tools: [Read, Write, Edit, Bash, Grep, Glob, Agent, ...], mcp: "动态连接外部工具", memory: "CLAUDE.md (项目级 + 用户级)", session: "JSONL 持久化", compaction: "6 层压缩管线", permission: "4 级权限控制", }
🧬
Context 维度
Context 工程成为显学。Claude Code 建立了 6 层压缩管线：Tool Result Budget → Snip Compact → Microcompact → Context Collapse → Auto-compact → Reactive Compact。/context 命令让用户首次看见 context 占用。Prompt Caching 也在此阶段引入，降低重复 context 的成本。
~100Ktokens 管理
解决的问题开箱即用的 agent runtime，开发者无需自建 loop
核心局限Compaction 是不可逆的——压缩后原始信息从 context 永久消失

Phase 52025.10 — 2025.12
+ Skills / 渐进式上下文披露
Agent = Model + Prompt + Tools + MCP + Skills

Skills 引入了"渐进式上下文披露"模式：注册时只注入 name + description（几行文字），触发时才展开完整指令和脚本。20 个 Skills 只占几百 token，但能按需加载海量专业能力。2025.12.18 作为 Agent Skills 开源标准发布。

2025.10.17Claude Skills 在 Claude Code v2.0.22 中推出
2025.12.18Agent Skills 作为开源标准发布
// 渐进式上下文披露 — 用时才加载 // 注册阶段: 只有几行描述 pdf: "Extract and analyze text from PDF documents" // 触发阶段: Claude 判断需要 → tool_use Skill({ command: "pdf" }) // 展开阶段: 完整指令 + 脚本路径注入 context → SKILL.md 全文 + extract_text.py 路径 // 20 个 MCP 工具: ~50,000 tokens // 20 个 Skills: ~ 500 tokens (100x 节省)
⚡
Context 维度
Context 膨胀问题的结构化解决。Skills 用"注册轻量 + 按需展开"模式，将 context 占用从 O(n×工具定义) 降到 O(n×描述)。同时 Tool Search 动态加载 MCP 工具，不再一次性全量注入。Context 从"静态填充"变为"动态装配"。
~500tokens 注册
解决的问题在不膨胀主 prompt 的前提下无限扩展能力
核心局限仍需自建 harness 基础设施来运行 Agent

Phase 62026.04
Managed Agents : 配置即 Agent
Agent = Config Object (model + prompt + tools + MCP + skills)

Managed Agents 将两年的演化成果统一为一个 API 配置对象。开发者只需定义 Agent 的组成（模型、提示、工具、MCP、Skills），Anthropic 托管运行全部基础设施。Brain/Hands/Session 三层解耦，按需弹性扩展。

2026.04.08Claude Managed Agents 公共 Beta 发布
// Agent 作为一等配置对象 const agent = createAgent({ model: "claude-sonnet-4-6-20250514", system_prompt: "You are a senior engineer...", tools: ["bash", "file", "web_search"], mcp_servers: [{ url: "..." }], skills: ["pdf", "deploy"], }); // 托管运行 — 无需自建任何基础设施 const session = startSession({ agent, env }); session.sendEvent("Refactor the auth module"); session.streamEvents(); // SSE 流式响应
🗃️
Context 维度
Session ≠ Context Window。Session 成为持久化 context 对象——append-only event log，通过 getEvents() 按位置切片查询。Harness 可以回退到任意时刻重读上下文。Context 管理策略（compaction、caching）由 Harness 决定，Session 只保证数据不丢。
∞可回溯
解决的问题配置即 Agent，无需自建 harness、sandbox、session 管理
架构创新Brain/Hands/Session 解耦，p50 TTFT 下降 60%，p95 下降 >90%

Phase 1-2 : 无管理期

开发者手动拼接消息数组，没有自动裁剪。200K 窗口是硬天花板，超出即报错。

messages[] 200K limit

Phase 3 : Context 膨胀危机

MCP 工具定义注入 system prompt，连接越多 Server 越占空间。用户开始意识到需要"context 预算"。

tool schema static injection

Phase 4 : Compaction 管线

Claude Code 建立 6 层压缩：Tool Result Budget → Snip → Microcompact → Context Collapse → Auto-compact → Reactive Compact。Prompt Caching 减少重复 context 成本。

autoCompact() prompt caching /context

Phase 5 : 渐进式披露

Skills 只注入描述（~25 token），触发时才展开全文。Tool Search 动态加载 MCP 工具。Context 从"预分配"变为"按需装配"。

lazy loading tool search 100x saving

Phase 6 : Session as Context

Session ≠ Context Window。append-only event log 持久化全部交互。Harness 通过 getEvents() 按需切片读取，可回溯任意时刻。Compaction 只影响 context 视图，不影响 session 数据。

getEvents() session log rewind

驱动演化的问题链

每一层的添加都源于上一层留下的缺口

新增层因为上一层的问题
+ Tools←模型只能输出文字，无法采取行动
+ MCP←工具连接没有标准，每个集成都是自定义开发
+ Harness (Claude Code)←每个开发者都在重复造 agent loop 的轮子
+ Skills←MCP 工具定义太大，撑爆 context window
+ Managed Agents←自建 harness + sandbox + session 管理太复杂
