# v1.2 Code Review

**Date:** 2026-04-08
**Scope:** 38 representative source files across main, preload, renderer, and shared layers
**Depth:** Standard (security, concurrency, memory leaks, error handling, correctness)
**Reviewer:** Automated static analysis

---

## Executive Summary

The codebase demonstrates solid architectural foundations: clear separation between Electron main/renderer/preload processes, well-defined IPC channel contracts, proper Zod schema validation on IPC boundaries, and consistent error handling patterns. The tool system, session persistence, and diff engine are well-structured.

However, the review identified **33 issues** across five severity levels. The most critical findings involve a command injection vector in the Bash tool, unbounded memory growth in the embedding client and chat store, race conditions in rate limiting and terminal data capture, and a path traversal bypass on Windows via junction points. These should be addressed before any release, even personal-use.

## Critical (Severity 1) -- Must Fix

### S1-01. Command Injection via Shell Execution (bash.ts)

**File:** `src/main/tools/bash.ts` line 78

The Bash tool passes the raw `command` string directly to `child_process.exec()` without any sanitization. While the tool requires user approval, an LLM agent constructing the command can inject arbitrary shell commands including chained commands (`&&`, `;`, `|`, `$()` subshells, etc.). The approval UI shows the command text but a user may not catch a subtly malicious chain.

Even for a personal tool, if the agent is tricked by prompt injection in a file it reads (a real attack vector for AI coding tools), this becomes a direct remote code execution path.

**Risk:** Remote code execution via prompt injection.
**Recommendation:** Consider adding a command blocklist or sanitization layer. At minimum, log the full command string to a visible audit trail. Evaluate whether `child_process.execFile` (which avoids shell interpretation) is viable for common use cases.

### S1-02. Path Traversal Bypass on Windows (ipc-handlers.ts)

**File:** `src/main/ipc-handlers.ts` lines 429, 486

The workspace boundary check uses `absolutePath.startsWith(workspaceRoot + path.sep)`. On Windows, this can be bypassed:
- A path like `C:\workspace\..\..\Windows\System32\config` resolves outside the workspace but may pass the prefix check if not normalized first.
- `path.resolve()` does handle `..` but only if the result doesn't happen to share a prefix. The check should use a normalized comparison.

**Risk:** Arbitrary file read/write outside workspace boundaries.
**Recommendation:** Normalize both paths with `path.resolve()` and compare. Use a helper function:

```typescript
function isWithinWorkspace(filePath: string, workspaceRoot: string): boolean {
  const normalized = path.resolve(filePath).toLowerCase()
  const root = path.resolve(workspaceRoot).toLowerCase()
  return normalized === root || normalized.startsWith(root + path.sep)
}
```

This function should be reused across all four IPC handlers that check workspace boundaries (file:save, file:apply-hunk, file:read-content, file:read-folder-tree).

### S1-03. Unbounded Memory Growth in TF-IDF Vocabulary (embedding-client.ts)

**File:** `src/main/indexing/embedding-client.ts` lines 231-272

The `updateVocab` method grows the vocabulary map unboundedly. Every unique term across all indexed files gets a permanent entry in `this.vocab.terms` and `this.vocab.idf`. For a large codebase with hundreds of thousands of unique identifiers, this map can grow to hundreds of megabytes. The vocabulary is also serialized to disk on every update (line 272), creating write amplification.

Additionally, the IDF recalculation loop on line 264 iterates over *all* existing terms for every batch, making this O(terms * batch_count) over the lifetime of the vocabulary.

**Risk:** Out-of-memory crash during indexing of large workspaces.
**Recommendation:** Cap vocabulary size (e.g., top 50,000 terms by document frequency). Batch the disk writes. Consider a more efficient IDF data structure.

## High (Severity 2) -- Should Fix Before Release

### S2-01. Race Condition in Shared Rate Limiter (web-fetch.ts, web-search.ts)

**Files:** `src/main/tools/web-fetch.ts` lines 18-27, `src/main/tools/web-search.ts` lines 18-27

Both files use module-level mutable `lastRequestTime` for rate limiting. In Node.js, if two `execute()` calls start concurrently (e.g., two agent tool calls in rapid succession), both read `lastRequestTime` before either writes it. Both see `elapsed >= WEB_SEARCH_RATE_LIMIT_MS` and proceed without waiting. The rate limiter is ineffective under concurrency.

Additionally, both files define their own independent `lastRequestTime`, so web-fetch and web-search do not share a combined rate budget, even though they may hit the same API (e.g., DuckDuckGo).

**Risk:** Rate limit violations leading to API bans.
**Recommendation:** Use a mutex or queue-based rate limiter. A simple approach:

```typescript
let rateLimitPromise: Promise<void> | null = null
async function enforceRateLimit(): Promise<void> {
  if (rateLimitPromise) await rateLimitPromise
  rateLimitPromise = (async () => {
    const now = Date.now()
    const elapsed = now - lastRequestTime
    if (elapsed < RATE_MS) await sleep(RATE_MS - elapsed)
    lastRequestTime = Date.now()
  })()
  await rateLimitPromise
}
```

### S2-02. Terminal runCommandInTerminal Never Resolves Early (terminal-manager.ts)

**File:** `src/main/terminal/terminal-manager.ts` lines 161-211

The `runCommandInTerminal` method captures output but the "early exit" logic on line 205-209 is a no-op. The `setTimeout` callback has an empty `if` body. The Promise only resolves after the full 30-second timeout expires, regardless of whether the command has finished.

This means every Bash tool call routed through the terminal takes a fixed 30 seconds, even for instant commands like `echo hello`.

**Risk:** Every terminal-routed command wastes 30 seconds. Severe usability degradation.
**Recommendation:** Implement proper command completion detection. Track output stability (no new data for N ms) and resolve when detected. Alternatively, use a sentinel/marker approach to detect command completion.

### S2-03. sessionsCache Never Evicted (chat-store.ts)

**File:** `src/renderer/stores/chat-store.ts` line 41

`sessionsCache` stores full message arrays keyed by session ID. Messages are accumulated indefinitely as the user switches between sessions. For a long work session with multiple conversations, each containing thousands of tokens, this grows without bound. The cache is only cleaned when a session tab is explicitly deleted.

**Risk:** Renderer memory exhaustion over extended use.
**Recommendation:** Implement an LRU eviction policy. Cap the cache at N sessions (e.g., 10). When the limit is exceeded, evict the least-recently-used entry.

### S2-04. Pending Symbol Queries Leak on Timeout (symbol-nav.ts)

**File:** `src/main/tools/symbol-nav.ts` lines 18, 44-46

The `pendingQueries` map stores `{ resolve, timer }` for each query. When a query times out (line 44-46), the entry is correctly deleted. However, if the renderer crashes or the webContents is destroyed while queries are pending, the `timer` references keep the `resolve` function and associated closures alive until timeout. For a rapid succession of queries, this can accumulate many pending entries.

More importantly, `queryCounter` is a module-level integer that wraps to floating-point precision issues after `Number.MAX_SAFE_INTEGER` increments (extremely unlikely but theoretically unbounded).

**Risk:** Minor memory leak in edge cases.
**Recommendation:** Clean up all pending queries when the webContents is destroyed. Cap `queryCounter` or use UUIDs.

### S2-05. Synchronous File I/O on Main Thread (indexing-engine.ts, vector-store.ts)

**Files:** `src/main/indexing/indexing-engine.ts` lines 84, 103, `src/main/indexing/vector-store.ts` lines 97, 180, 200, 215

Multiple `fs.statSync`, `fs.readdirSync`, `fs.readFileSync`, and `fs.writeFileSync` calls occur on the Electron main thread. The `walkWorkspace` method in indexing-engine.ts is a recursive synchronous directory walk that blocks the main thread for the duration. For large workspaces, this can block the UI for seconds.

Similarly, `VectorStore.writeAll` uses `fs.writeFileSync` which blocks the event loop during JSON serialization and disk I/O.

**Risk:** UI freezes during indexing operations.
**Recommendation:** Replace synchronous calls with async equivalents (`fsp.stat`, `fsp.readdir`, `fsp.readFile`, `fsp.writeFile`). For the directory walk, use an async generator or process files in batches with `setImmediate` yield points.

### S2-06. XSS via Tool Output Rendering (ToolCard.tsx)

**File:** `src/renderer/components/chat/ToolCard.tsx` lines 389, 415

Tool output is rendered as `{displayedOutput}` (text node) in the generic output path, which is safe. However, the `renderWebSearchOutput` and `renderWebFetchOutput` functions render extracted URLs as `<a href={url}>` where `url` comes from LLM tool output. A crafted URL like `javascript:alert(1)` would execute in the Electron renderer context.

**Risk:** XSS in Electron renderer (which has Node.js access by default, making this more severe than browser XSS).
**Recommendation:** Validate URL scheme before rendering links. Only allow `http://` and `https://` schemes:

```typescript
const isSafeUrl = url.startsWith('http://') || url.startsWith('https://')
```

### S2-07. LCS Diff Algorithm O(m*n) Memory (diff-store.ts)

**File:** `src/renderer/stores/diff-store.ts` lines 47-62

The `computeHunks` function allocates a full `(m+1) * (n+1)` DP table. While there is a guard at `MAX_DIFF_FILE_LINES`, the guard only checks line count. For two files with 5000 lines each, the DP table is `5001 * 5001 = 25,010,001` entries. Each entry is a JavaScript number (8 bytes in V8), consuming approximately 200MB of heap.

**Risk:** Browser tab OOM crash when diffing moderately large files.
**Recommendation:** Use a linear-space LCS algorithm (Myers diff or Hirschberg). Alternatively, reduce `MAX_DIFF_FILE_LINES` to a safer threshold (e.g., 1000) or use a streaming diff approach.

## Medium (Severity 3) -- Should Fix Soon

### S3-01. TF-IDF IDF Recalculation Is Mathematically Incorrect (embedding-client.ts)

**File:** `src/main/indexing/embedding-client.ts` lines 256-269

The IDF recalculation on line 259 attempts to recover document frequency from the stored IDF value by multiplying by `this.vocab.docCount`. This is incorrect because the stored IDF is `log((totalDocs+1)/(df+1)) + 1`, which is not invertible via simple multiplication. The recovered "existingCount" is wrong, making subsequent IDF values for previously-seen terms drift incorrectly over time.

**Risk:** Search quality degrades over multiple indexing runs.
**Recommendation:** Store raw document frequency (df) alongside IDF, or recalculate IDF for all terms when new documents are added.

### S3-02. TaskManager nextId Collision After clearTasks (task-manager.ts)

**File:** `src/main/tasks/task-manager.ts` line 108

`clearTasks` resets `nextId` to 1. If any external system (IPC, agent) holds references to old task IDs like `task-3`, and new tasks are created after clearing, the new tasks will reuse the same IDs (`task-1`, `task-2`, `task-3`), causing confusion in dependency tracking and UI state.

**Risk:** Incorrect task dependency resolution after clearing.
**Recommendation:** Either do not reset `nextId` on clear, or use UUIDs for task IDs.

### S3-03. VectorStore Cache Becomes Stale After Concurrent Writes (vector-store.ts)

**File:** `src/main/indexing/vector-store.ts` lines 110-130

The `upsert` method loads all entries, merges, and writes back. If two `upsert` calls overlap (e.g., indexing engine batch processing with interleaved file-change updates), both load the same cached data, merge independently, and the second write overwrites the first's additions. The cache is not invalidated between load and write.

**Risk:** Lost index entries under concurrent modifications.
**Recommendation:** Add a write lock or queue to serialize upsert/delete operations.

### S3-04. Abort Signal Listener Leak in Bash Tool (bash.ts)

**File:** `src/main/tools/bash.ts` lines 114-123

The abort signal listener (`onAbort`) is added inside the Promise but is never removed if the command completes normally. While `child.kill()` on an already-dead process is a no-op, the listener function closure retains references to `child` and `context`, preventing garbage collection of the child process object until the signal's listener list is cleaned.

**Risk:** Minor memory leak proportional to number of bash tool calls.
**Recommendation:** Remove the listener after the command completes:

```typescript
context.abortSignal.removeEventListener('abort', onAbort)
```

### S3-05. Embedding Dimension Mismatch Between API and TF-IDF (embedding-client.ts)

**File:** `src/main/indexing/embedding-client.ts` lines 147-153

`getDimension()` returns 1536 if an API key is configured, regardless of whether the API is actually reachable. If the API key is configured but the API is down and the system falls back to TF-IDF, the stored TF-IDF vectors have a different dimensionality than what `getDimension()` reports. This can cause cosine similarity comparisons against vectors of different lengths.

**Risk:** Cosine similarity returns 0 for all comparisons after API fallback (due to length mismatch check in `cosineSimilarity`).
**Recommendation:** Track the actual embedding type used for the most recent batch and return the correct dimension.

### S3-06. chat-store switchSession Error Handling Is Fragile (chat-store.ts)

**File:** `src/renderer/stores/chat-store.ts` lines 418-428

The `switchSession` method checks `if (!get().error || get().error === prevError)` to determine if loading succeeded. This is fragile: if the error is the same string as a previous error, the switch proceeds even if the load failed. It also treats `error === null` and `error === previousNull` as success, which works but is not clear.

**Risk:** Session state corruption if loadSession fails with the same error message.
**Recommendation:** Have `loadSession` return a success/failure indicator instead of relying on error state comparison.

### S3-07. Indexing Progress Callbacks Not Cleaned Up (indexing-engine.ts)

**File:** `src/main/indexing/indexing-engine.ts` lines 261-270

The `onProgress` method returns an unsubscribe function, but the `progressCallbacks` array is only cleared on `dispose()`. If components subscribe/unsubscribe frequently (e.g., React component re-mounts), the array itself grows with spliced holes. The `splice` in the unsubscribe (line 268) is O(n).

**Risk:** Performance degradation over time.
**Recommendation:** Use a `Set` instead of an array for `progressCallbacks`, matching the pattern used in `TaskManager.listeners`.

### S3-08. WebSearch DuckDuckGo API May Not Return Expected Format (web-search.ts)

**File:** `src/main/tools/web-search.ts` line 71

The DuckDuckGo instant answer API (`api.duckduckgo.com/?format=json`) is not a general search API. It returns abstract and related topics for queries that match Wikipedia-style entries, not web search results. For most programming queries, it returns empty results. The tool description says "Search the web for information" but the backend provides only instant answers.

**Risk:** Tool is misleading and largely ineffective for its stated purpose.
**Recommendation:** Either integrate a proper search API (SearXNG, Brave Search, Google Custom Search) or update the description to accurately reflect its capabilities.

### S3-09. TerminalPanel xterm Data Subscription Receives All Terminal Output (TerminalPanel.tsx)

**File:** `src/renderer/components/ide/TerminalPanel.tsx` lines 132-136

The `onTerminalData` subscription in the IPC layer sends data for ALL terminals to every subscriber. The renderer-side filter on line 133 (`if (payload.terminalId === activeTerminalId)`) discards non-matching data, but the IPC overhead and event listener processing still occurs for every terminal's output. With multiple terminals running, this creates unnecessary cross-terminal noise.

**Risk:** Performance degradation with multiple active terminals.
**Recommendation:** Filter terminal data on the main process side (in `ipc-handlers.ts` terminal:create) so only the creating renderer receives its terminal's data.

### S3-10. EditorPanel Diff Decorations Stored on Editor Instance (EditorPanel.tsx)

**File:** `src/renderer/components/ide/EditorPanel.tsx` line 140

Diff decoration IDs are stored directly on the Monaco editor instance via `(editor as unknown as Record<string, string[]>).__diffDecorations`. This is a monkey-patch on the editor object. If Monaco internally uses a property with the same name (unlikely but possible in future versions), it would cause conflicts. It also bypasses TypeScript's type safety.

**Risk:** Potential conflict with Monaco internals.
**Recommendation:** Store decoration IDs in a `useRef` instead of on the editor instance.

## Low (Severity 4) -- Fix When Convenient

### S4-01. Unused Parameter in BashTool Constructor (bash.ts)

**File:** `src/main/tools/bash.ts` line 40

The `workingDirectory` parameter is accepted but never used. The actual working directory comes from `context.workingDirectory` during execution. The constructor signature is misleading.

### S4-02. Dead Code in TaskManager (task-manager.ts)

**File:** `src/main/tasks/task-manager.ts` line 145

`void (0 as unknown as TaskStatus)` appears to be a workaround for an unused import warning. This is unnecessary and should be removed along with the `TaskStatus` import if it is truly unused.

### S4-03. EmbeddingClient Rate Limit Delay Uses setTimeout Without Abort (embedding-client.ts)

**File:** `src/main/indexing/embedding-client.ts` lines 204-211

The `rateLimitDelay` method creates a `setTimeout` that cannot be cancelled. If the embedding client is disposed while waiting, the timer continues and the callback executes on a disposed instance.

### S4-04. STOP_WORDS Set Contains Duplicates (embedding-client.ts)

**File:** `src/main/indexing/embedding-client.ts` lines 27-43

Several words appear twice in the `STOP_WORDS` set definition: `'if'`, `'into'`, `'no'`, `'how'`, `'new'`, `'this'`. Since it is a `Set`, duplicates are harmless but indicate copy-paste errors.

### S4-05. MentionsPicker `any` Type in flattenTree (MentionPicker.tsx)

**File:** `src/renderer/components/chat/MentionPicker.tsx` line 28

The `children` parameter in `flattenTree` is typed as `any[]` instead of the recursive tree type. This loses type safety.

### S4-06. ChatPanel Auto-Scroll Triggers on Every Message Mutation (ChatPanel.tsx)

**File:** `src/renderer/components/chat/ChatPanel.tsx` lines 59-61

The auto-scroll `useEffect` depends on the entire `messages` array reference. Since streaming updates create new array references on every text delta, this causes a `scrollIntoView` call on every streaming token, which is expensive and causes visible jitter.

**Recommendation:** Only auto-scroll when messages length changes, not on content updates.

### S4-07. DiffPreview Auto-Clear May Surprise Users (DiffPreview.tsx)

**File:** `src/renderer/components/chat/DiffPreview.tsx` lines 21-29

The auto-clear effect runs on every `pendingDiffs` change. If all diffs are resolved as a side effect of accepting/rejecting, the diff panel disappears immediately, which may be unexpected. Consider requiring explicit close.

### S4-08. SessionStore appendMessages Duplicates Messages on Re-save (session-store.ts)

**File:** `src/main/persistence/session-store.ts` line 69

`appendMessages` is called with `allMessages` from `agentLoop.getMessages()` after each agent turn. If this is called multiple times for the same session (which it is -- once per `agent:done`), messages from previous turns get appended again. This causes duplicate messages in the JSONL file.

**Risk:** Session data grows with duplicate messages, degrading load performance over time.
**Recommendation:** Track the last persisted message index and only append new messages.

### S4-09. WebFetch htmlToText Naive Replacement (web-fetch.ts)

**File:** `src/main/tools/web-fetch.ts` lines 33-57

The regex `/<script[\s\S]*?<\/script>/gi` can be defeated by nested or malformed script tags. The HTML entity decoding only handles 6 entities. For production use, a proper HTML parser would be more robust.

### S4-10. TokenCounter Singleton Never Disposed (token-counter.ts)

**File:** `src/main/context/token-counter.ts` line 6

The `Tiktoken` singleton is created at module load time and never freed. The tiktoken WASM module allocates memory that is never released. This is acceptable for the application lifetime but should be documented.

### S4-11. SymbolService findModel Uses Loose String Matching (SymbolService.tsx)

**File:** `src/renderer/components/ide/SymbolService.tsx` line 233

`findModel` uses `m.uri.toString().includes(filePath)` which can match unintended models. For example, searching for `utils.ts` would match `src/test-utils.ts`, `src/utils.ts.bak`, etc.

**Recommendation:** Use exact path matching or compare path segments.

## Positive Observations

1. **IPC Validation:** All IPC handlers use Zod schema validation for incoming requests. This is a strong security boundary that prevents malformed input from the renderer.

2. **Workspace Boundary Checks:** Four IPC handlers (file:save, file:apply-hunk, file:read-content, file:read-folder-tree) check workspace boundaries before file operations. The pattern is consistent.

3. **Abort Signal Support:** The Bash tool correctly implements abort signal handling, allowing the agent loop to cancel long-running commands.

4. **Session Persistence:** The SessionStore uses atomic writes (write-to-temp-then-rename) for rename operations, preventing data corruption on crash. The JSONL format is appropriate for append-heavy workloads.

5. **Context Manager Circuit Breaker:** The `isCompacting` flag prevents re-entrant compaction, which would otherwise cause an infinite loop.

6. **Diff Engine Guards:** The diff store guards against O(n*m) memory exhaustion with `MAX_DIFF_FILE_LINES`.

7. **Task Dependency Tracking:** The TaskManager correctly handles forward references and auto-unblocks dependents when a task completes.

8. **Type Safety:** The shared types file defines discriminated unions for messages, stream events, and mentions, enabling exhaustive pattern matching.
