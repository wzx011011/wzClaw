<task type="auto">
  <name>Task 1: Configure electron-builder for Windows NSIS installer</name>
  <files>electron-builder.yml, build/icon.ico</files>
  <action>
1. Create the `build/` directory if it does not exist.

2. Generate a placeholder application icon. Use ImageMagick or a Node.js script to create a simple 256x256 ICO file at `build/icon.ico`. If neither ImageMagick nor a suitable npm package is available, create a minimal valid ICO file using raw bytes in a Node.js one-liner. The icon does not need to be beautiful -- a solid-color square is fine for a personal tool. Alternatively, check if electron-vite's template already provides a default icon in build/ and use that.

3. Update `electron-builder.yml` with complete Windows packaging configuration:
```yaml
appId: com.wzxclaw
productName: wzxClaw
directories:
  buildResources: build
  output: dist
files:
  - '!**/.vscode/*'
  - '!src/*'
  - '!electron.vite.config.{js,ts,mjs,cjs}'
  - '!{.eslintignore,.eslintrc.cjs,.prettierignore,.prettierrc.yaml,dev-app-update.yml,CHANGELOG.md,README.md}'
  - '!{tsconfig.json,tsconfig.node.json,tsconfig.web.json}'
  - '!{.env,.env.*}'
  - '!**/__tests__/*'
  - '!**/*.test.*'
  - '!**/*.spec.*'
  - '!.planning/*'
win:
  icon: build/icon.ico
  target:
    - target: nsis
      arch:
        - x64
nsis:
  oneClick: false
  perMachine: false
  allowToChangeInstallationDirectory: true
  deleteAppDataOnUninstall: false
  shortcutName: wzxClaw
```

Key decisions:
- NSIS installer (not portable) because user wants an installable app per CONTEXT.md
- `oneClick: false` so user can choose install directory
- `deleteAppDataOnUninstall: false` to preserve settings/API keys stored by safeStorage
- Exclude `.env` files and test files from the package to keep it lean and avoid leaking API keys
- Exclude `.planning/` directory -- development-only
- x64 only since this is a personal tool on Windows 10+

4. Verify the existing `build:win` script in package.json is correct: `"build:win": "npm run build && electron-builder --win"`. If it differs, update it.

5. Run `npm run build` (electron-vite build only, NOT the full package) to verify the build compiles without errors. Do NOT run `npm run build:win` -- that is for Plan 02.

<task type="auto">
  <name>Task 2: Verify build output structure and fix packaging issues</name>
  <files>electron-builder.yml, package.json</files>
  <action>
1. Run `npm run build:unpack` which builds and creates an unpacked directory without creating the full installer. This is faster than the full NSIS build and reveals packaging issues.

2. After the unpack build completes, inspect the output directory structure:
```bash
ls dist/win-unpacked/
ls dist/win-unpacked/resources/
```

3. Verify the unpacked build contains:
   - `wzxClaw.exe` (the main executable)
   - `resources/app.asar` (the bundled application code)
   - The exe launches (try running it briefly with a timeout)

4. Common issues to check and fix:
   - If `app.asar` is missing or empty, the files exclusion pattern may be too aggressive. Check `out/main/`, `out/preload/`, `out/renderer/` exist after electron-vite build.
   - If the exe crashes immediately, check that `main` field in package.json points to `./out/main/index.js` (already set).
   - If Monaco Editor assets are missing, electron-vite may need the renderer output to include static assets. Check `out/renderer/` has `index.html` and JS bundles.

5. If any issues are found, fix electron-builder.yml or package.json and re-run `npm run build:unpack` to verify.

6. Do NOT delete the dist/ output -- Plan 02 will use it for E2E verification.
  </action>

  </verify>

    - dist/win-unpacked/resources/app.asar exists
    - No critical build errors
    - Package size is reasonable (< 500MB)
  </done>
